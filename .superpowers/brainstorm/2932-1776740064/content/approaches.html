Three architectural approaches for core-shell

How the shell and its extensions relate to each other. Pick the architecture, everything else follows.

RECOMMENDED
1

Thin React Shell — Extensions Are Components

The shell is a React layout engine. Extensions export React components that mount into named slots. No process separation. Everything in one renderer.

/* Electron Renderer (one Chromium process) */
React Shell
├─ ext-amplifier-canvas → mounts Canvas slot
├─ ext-git-panel → mounts Sidebar slot
└─ ext-status-items → mounts StatusBar slot
/* Extension author writes: */
export default { manifest, Canvas, SidebarPanel }

Pros

  • Zero friction — one export, done
  • Hot reload works automatically
  • Share React context, Zustand stores
  • Cloud-portable (no Node.js APIs in extensions)

Trade-offs

  • Extension crash → shell crash
  • No process isolation for marketplace
  • (fixable later with sandbox layer)
2

Extension Host + WebViews — VS Code Model

Separate Node.js extension host process. Canvas content renders in sandboxed WebContentsViews. Full process isolation. High fidelity to VS Code's architecture.

Electron Main Process
├─ Extension Host (Node.js)
├─ ext-A.js ↔ IPC ↔ shell API
└─ ext-B.js ↔ IPC ↔ shell API
└─ WebContentsViews (sandboxed)
├─ ext-A canvas (isolated renderer)
└─ ext-B canvas (isolated renderer)

Pros

  • True process isolation
  • Extension crash is contained
  • Extensions can use Node.js APIs

Trade-offs

  • 6–12 months to build well
  • IPC for every UI interaction
  • Big friction for extension authors
  • Harder to port to cloud
3

Hybrid — Trusted Components + Optional Sandbox

First-party extensions (amplifier-canvas, your own tools) run as React components in the shell. Third-party marketplace extensions get sandboxed in WebContentsViews. Trust level in manifest.

React Shell (Renderer)
├─ trusted: amplifier-canvas ← inline
├─ trusted: git-panel ← inline
└─ sandboxed: 3rd-party-ext ← WebView

Pros

  • No friction for your own work (trusted)
  • Isolation when marketplace opens
  • Gradual migration path

Trade-offs

  • Two extension APIs to maintain
  • Trust model needs careful design
  • More complex than Approach 1

Approach 1 is the right call for now — you can add the sandbox layer from Approach 3 when a public marketplace becomes real. Don't build isolation you don't need yet.
