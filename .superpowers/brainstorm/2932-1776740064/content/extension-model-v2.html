Section 3 — Extension Model

What you write to add a new canvas. The complete contract — no more, no less.

An extension is one folder with one entry point

extensions/amplifier-canvas/index.tsx — exports exactly one thing
export const manifest = {
  id:          'amplifier-canvas',
  displayName: 'Amplifier Canvas',
  version:     '0.1.0',

  contributes: {
    activityRail: { icon: SessionsIcon, title: 'Sessions' },
    sidebar:      SessionListSidebar,
    canvas:       CanvasMain,            // ← the primary UI
    panel:        { title: 'Terminal', component: TerminalPanel },
    statusBar:    { position: 'left', component: StatusItem },
  },
}

Components are passed as values — not strings, not paths. No indirection. The shell imports your manifest and mounts whatever you put in each slot.

What the shell gives your extension — useShell()

Available inside any extension component
const shell = useShell()

// IPC bridge → main process (file system, PTY)
await shell.ipc.readFile(path)
await shell.ipc.writeFile(path, content)
const unsub = shell.ipc.watchDir(path, callback)
const pty   = await shell.ipc.spawn(cmd, args)

// Navigation
shell.openTab(extensionId)
shell.setTitle(text)

// Command palette
shell.registerCommand('canvas.clearSessions', handler)
shell.executeCommand('workbench.reload')

// Per-extension state (isolated, persisted by shell)
const [state, setState] = useExtensionStore({ sessions: [], active: null })

Extensions never import from Electron or Node directly. Everything goes through this bridge — which is why extension code is already cloud-portable.

Loading on startup

🔍
1. Discover
Scan extensions/ for subfolders with an index.tsx
📦
2. Import
Dynamic import() of the manifest. Components stay lazy until activated.
🗂
3. Register
Contributions added to slot registry. Activity icon appears. Commands registered.
🛡
4. Mount
Component mounts inside a React error boundary. Crash = that slot only.
